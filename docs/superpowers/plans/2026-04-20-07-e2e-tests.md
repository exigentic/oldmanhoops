# OldManHoops E2E Tests Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Playwright end-to-end coverage for the five flows listed in the spec: visitor view, signup form, RSVP-from-site (authenticated member), RSVP-from-email (HMAC link), and settings.

**Architecture:** Each spec file drives a browser through a real flow against the local dev server and local Supabase stack. A shared `fixtures.ts` extends Playwright's `test` object with (a) a `cleanToday` fixture that ensures today's game row exists and is in scheduled state before each test, and (b) an `authedUser` fixture that creates a unique auth user, establishes a session via `supabase.auth.admin.generateLink` → `page.request.post('/api/auth/verify')`, yields the user's email + id to the test, and deletes the user on teardown. Tests run serially (matching Jest) because they share a single local Supabase DB.

**Tech Stack:** Playwright 1.59, @supabase/supabase-js admin client, existing `lib/hmac` for token generation. No new runtime dependencies.

**Prerequisites:**
- Plans 1–6 complete.
- Local Supabase stack running (`supabase start`) with migrations applied.
- `.env.local` populated; the dev server at `http://localhost:3000` must point at the local Supabase.

## Design Decisions

**Session injection via generateLink + page.request.** Playwright cannot click real magic-link emails without intercepting the inbox. Instead the fixture creates a user via the admin client, calls `admin.auth.admin.generateLink({ type: 'magiclink', email })`, takes the returned `email_otp` (6-digit code Supabase would have emailed), and POSTs it to our existing `/api/auth/verify` endpoint via `page.request.post`. That path consumes the OTP, sets the Supabase session cookies on the browser context, and leaves the page ready to navigate authenticated. No new test-only endpoint, no email-box polling.

**Serial execution.** Playwright's `fullyParallel: true` races on the shared local Supabase state just like Jest's default workers did. We set `fullyParallel: false` in `playwright.config.ts` so tests run one at a time per project (chromium and mobile still run as separate projects).

**Unique emails per test.** Every fixture-created user uses `e2e-${Date.now()}-${Math.random().toString(36).slice(2, 8)}@example.com` so nothing collides if a teardown is ever skipped.

**Today's game always scheduled.** The `cleanToday` fixture upserts today's `games` row to `status='scheduled'` before the test runs. It never deletes the row (other parallel-worker tests may depend on it) and does not clean up RSVPs the test created — teardown of the user cascades the RSVPs automatically via the foreign key.

**No email-box interception.** The signup spec stops at "the OTP form appeared" rather than completing an end-to-end signup, because finishing would require reading Supabase's inbucket for the invite OTP. That work is one level of infrastructure beyond v1. The rest of the session-requiring flows use the generateLink+verify shortcut.

**RSVP-from-email uses a directly-signed token.** The fixture's signer creates an HMAC token via `signToken` from `lib/hmac`, mirrors what the cron would have sent in the email, and drives the browser to `/api/rsvp?token=...&status=in&player_id=...&game_id=...`. The route verifies the token, upserts the RSVP, logs the user in, and redirects to `/?status=in`. No email is actually sent.

**No change to app code.** This plan adds tests only; zero modifications to `app/`, `lib/`, or API routes.

## Task 2: Test fixtures (`tests/e2e/fixtures.ts`)

**Files:**
- Create: `tests/e2e/fixtures.ts`

**Behaviour:**
- Extends Playwright's `test` with two fixtures:
  - `cleanToday` — an `auto` fixture that upserts today's `games` row to `scheduled` before each test
  - `authedUser` — creates a unique user, establishes a session on the page, yields `{ email, userId }`, deletes user on teardown
- Uses `createAdminClient` from `lib/supabase/admin`
- Loads `.env.local` via dotenv (Playwright's `webServer` inherits env from the shell, but our admin client reads `env.APP_TIMEZONE` at import time — simplest is to import `dotenv` at fixture top)

- [ ] **Step 1: Create the fixtures file**

Create `tests/e2e/fixtures.ts`:

```ts
import { config as loadEnv } from "dotenv";
import { resolve } from "node:path";
import { test as base, expect } from "@playwright/test";
import { DateTime } from "luxon";

// Load .env.local so dynamic imports of lib/env resolve at fixture init.
loadEnv({ path: resolve(process.cwd(), ".env.local") });

// Import lazily so env vars are populated before lib/env evaluates.
async function adminClient() {
  const mod = await import("../../lib/supabase/admin");
  return mod.createAdminClient();
}

function uniqueEmail(): string {
  const rand = Math.random().toString(36).slice(2, 8);
  return `e2e-${Date.now()}-${rand}@example.com`;
}

function todayInAppTz(): string {
  const zone = process.env.APP_TIMEZONE ?? "America/New_York";
  return DateTime.now().setZone(zone).toFormat("yyyy-MM-dd");
}

type AuthedUser = { email: string; userId: string };

type Fixtures = {
  cleanToday: void;
  authedUser: AuthedUser;
};

export const test = base.extend<Fixtures>({
  // Auto-applied: ensures today's game row exists and is scheduled.
  cleanToday: [
    async ({}, use) => {
      const admin = await adminClient();
      const today = todayInAppTz();
      await admin
        .from("games")
        .upsert(
          { game_date: today, status: "scheduled" },
          { onConflict: "game_date" }
        );
      await use();
    },
    { auto: true },
  ],

  authedUser: async ({ page }, use) => {
    const admin = await adminClient();
    const email = uniqueEmail();

    // Create user (handle_new_user trigger inserts the players row).
    const { data: created, error: createErr } = await admin.auth.admin.createUser({
      email,
      email_confirm: true,
      user_metadata: { name: `E2E ${email.slice(4, 10)}` },
    });
    if (createErr) throw createErr;
    const userId = created.user!.id;

    // Generate a magic link and extract the one-time OTP.
    const { data: link, error: linkErr } = await admin.auth.admin.generateLink({
      type: "magiclink",
      email,
    });
    if (linkErr) throw linkErr;
    const otp = link.properties?.email_otp;
    if (!otp) throw new Error("generateLink did not return email_otp");

    // Hand the OTP to our verify endpoint via the browser context so the
    // Supabase session cookies land on this page's context.
    const res = await page.request.post("/api/auth/verify", {
      data: { email, token: otp, type: "email" },
    });
    expect(res.ok(), `verify returned ${res.status()}`).toBeTruthy();

    await use({ email, userId });

    // Teardown — delete the user; rsvps cascade.
    await admin.auth.admin.deleteUser(userId);
  },
});

export { expect };
```

## Task 6: RSVP from email (`rsvp-from-email.spec.ts`)

**Files:**
- Create: `tests/e2e/rsvp-from-email.spec.ts`

**Behaviour:**
- Creates a user (via a minimal local helper — we don't use `authedUser` because we don't want the fixture to pre-log them in; the HMAC-link flow logs them in on its own)
- Creates today's game row (auto via cleanToday fixture)
- Reads today's game id from Supabase
- Signs an HMAC token for the user+game+status
- Navigates the browser to `/api/rsvp?token=X&status=in&player_id=Y&game_id=Z`
- Asserts the final URL is `/?status=in`
- Asserts the confirmation banner "You're In!" is visible
- Asserts a protected page (`/settings`) loads (proves session was established)

- [ ] **Step 1: Write the test**

Create `tests/e2e/rsvp-from-email.spec.ts`:

```ts
import { test, expect } from "./fixtures";
import { signToken } from "../../lib/hmac";

test("clicking an email link RSVPs the user, logs them in, and shows banner", async ({
  page,
}) => {
  // Dynamic imports so env is populated before lib/env.ts evaluates.
  const { createAdminClient } = await import("../../lib/supabase/admin");
  const { getToday } = await import("../../lib/date");

  const admin = createAdminClient();
  const email = `e2e-email-${Date.now()}@example.com`;

  // Create user (trigger inserts players row).
  const { data: created, error: createErr } = await admin.auth.admin.createUser({
    email,
    email_confirm: true,
    user_metadata: { name: "E2E Email" },
  });
  if (createErr) throw createErr;
  const userId = created.user!.id;

  try {
    // Fetch today's game row (cleanToday fixture already upserted it).
    const { data: game, error: gameErr } = await admin
      .from("games")
      .select("id")
      .eq("game_date", getToday())
      .single();
    if (gameErr) throw gameErr;

    const secret = process.env.HMAC_SECRET!;
    const token = signToken(
      {
        player_id: userId,
        game_id: game.id,
        status: "in",
        expires_at: Date.now() + 8 * 60 * 60 * 1000,
      },
      secret
    );

    const url = `/api/rsvp?token=${encodeURIComponent(token)}&status=in&player_id=${userId}&game_id=${game.id}`;
    await page.goto(url);

    // After all redirects, land on / with ?status=in
    await page.waitForURL(/\/\?status=in$/);

    // Confirmation banner visible
    await expect(page.getByText(/You're In!/i)).toBeVisible();

    // Navigate to /settings — if the session didn't stick we'd be bounced to /login
    await page.goto("/settings");
    await expect(page.getByRole("heading", { name: /^Settings$/i, level: 1 })).toBeVisible();
  } finally {
    await admin.auth.admin.deleteUser(userId);
  }
});
```

test("member updates their display name and toggles off email reminders", async ({
  page,
  authedUser,

  await expect(
    page.getByRole("heading", { name: /^Settings$/i, level: 1 })
